Understanding GitHub: Our Digital Library System

GitHub is like a digital library where many people can work together on books (projects) at the same time. Imagine a magical book that keeps track of every change ever made to it, who made the change, and when. That's what GitHub does for our knowledge.

Key Concepts

  • Repository (Repo): Think of this as our main library building. It contains all our books, documents, and materials in digital form. Our repository will hold everything from farming guides to meditation practices.
  • Fork: This is like making a complete copy of our library for yourself. When you fork our repository, you get your own version that you can modify without affecting the original. It's similar to having a branch of our school, but in digital form.
  • Commit: Every time you make a change - like adding a new farming technique or updating a meditation guide - you create a commit. It's like taking a snapshot of the library at that moment, recording exactly what changed.
  • Pull Request: When you want to share your changes with the main library, you make a pull request. It's like saying "I've improved this farming guide, would you like to include my changes in the main library?" The community can then review and discuss these changes.
  • Branch: Think of this as a separate workspace where you can experiment with new ideas without affecting the main collection. It's like having a draft version of a book that you're working on.
  • Merge: When a change is approved, it gets merged into the main collection. It's like taking the improved version of a book and placing it on the main library shelves.

GitHub एउटा डिजिटल पुस्तकालय जस्तै हो जहाँ धेरै मान्छेहरूले एकैसाथ पुस्तकहरू (प्रोजेक्टहरू) मा काम गर्न सक्छन्। एउटा जादुई पुस्तक कल्पना गर्नुहोस् जसले यसमा गरिएको हरेक परिवर्तन, कसले परिवर्तन गर्यो र कहिले गरियो भन्ने कुराको ट्र्याक राख्छ। GitHub ले हाम्रो ज्ञानको लागि यही काम गर्छ।

मुख्य अवधारणाहरू

  • रिपोजिटरी (रेपो): यसलाई हाम्रो मुख्य पुस्तकालय भवन जस्तै सोच्नुहोस्। यसमा हाम्रा सबै पुस्तकहरू, कागजातहरू र सामग्रीहरू डिजिटल रूपमा छन्।
  • फोर्क: यो तपाईंको लागि हाम्रो पुस्तकालयको पूर्ण प्रतिलिपि बनाउनु जस्तै हो। जब तपाईं हाम्रो रिपोजिटरी फोर्क गर्नुहुन्छ, तपाईंले आफ्नै संस्करण पाउनुहुन्छ।
  • कमिट: हरेक पटक तपाईंले परिवर्तन गर्दा - जस्तै नयाँ खेती प्रविधि थप्दा वा ध्यान मार्गदर्शन अपडेट गर्दा - तपाईंले कमिट सिर्जना गर्नुहुन्छ।
  • पुल रिक्वेस्ट: जब तपाईं आफ्ना परिवर्तनहरू मुख्य पुस्तकालयसँग साझा गर्न चाहनुहुन्छ, तपाईंले पुल रिक्वेस्ट बनाउनुहुन्छ।

How Our Knowledge System Works

Repository Access Levels

  • Core Maintainers (Staff and Senior Students)
    • Have direct write access to the main branch
    • Review and approve pull requests
    • Manage repository settings and access controls
    • Oversee the overall direction of the knowledge base
  • Regular Contributors (Students, Teachers, Alumni)
    • Fork the repository to their personal accounts
    • Submit changes through pull requests
    • Contribute to discussions on issues and improvements
    • Maintain their own versions with customizations
  • Public Contributors (Volunteers, Other Organizations)
    • Fork the repository for their own use
    • Submit suggestions through pull requests
    • Adapt the knowledge base for their needs
    • Share improvements back to the main repository

Contribution Workflow

  • Local Development
    • Contributors clone or fork the repository
    • Create a new branch for their changes
    • Make and test changes locally
    • Commit changes with clear, descriptive messages
  • Review Process
    • Submit pull request with detailed description
    • Core maintainers review changes
    • Community discussion and feedback
    • Iterative improvements based on feedback
  • Knowledge Distribution
    • Approved changes merge into main repository
    • Automatic deployment to nepal-web.net
    • Other organizations can fork and customize
    • Regular synchronization opportunities

Growth and Evolution

  • Forking and Customization
    • Organizations can fork the entire knowledge base
    • Adapt content to local needs and culture
    • Maintain independence while preserving core principles
    • Share improvements back through pull requests
  • Local Copies
    • Download complete repository for offline use
    • Regular synchronization when online
    • Resilient to internet connectivity issues
    • Support for remote or disconnected locations
